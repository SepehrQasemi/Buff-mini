# README Rewrite Report

## Why The Old README Was Wrong

The old README still described Buff-mini as an early MVP focused on Stage-0 through Stage-10 era infrastructure. That was materially misleading because the current repository now contains:
- late-stage evidence and validation semantics
- explicit exploration vs evaluation mode controls
- synthetic truth-lab detectability proof
- mechanism-family generation and audit layers
- candidate-specific ranking and funnel diagnostics
- data fitness, transfer intelligence, and campaign inventory stages
- PR-governed workflow assumptions

## What Was Rewritten

The README was rewritten completely to reflect the real current system, including:
- project overview and non-goals
- maturity state
- core principles
- architecture overview
- stage-group roadmap summary from Stage-74 through Stage-94
- run modes and interpretability semantics
- validation philosophy
- data philosophy and blocking semantics
- current known limitations
- quick start and key commands
- development workflow
- artifact guide
- blunt honest status section

## Why The New README Matches Reality Better

The new README is aligned with the actual code and artifacts because it:
- describes the repaired late-stage validation path instead of the MVP-only path
- states that no robust live edge is currently established
- states that evaluation-mode BTC/ETH campaigns are still blocked by continuity issues
- distinguishes exploratory outputs from evaluation-grade outputs
- points readers to the current stage runners and artifact files that actually exist in this repo

## Sections Removed Or De-Emphasized

The old README over-emphasized:
- Stage-0 feasibility framing
- early MVP-era claims
- older stage narratives as if they were the current story

Those sections were removed or reduced so the README now leads with the repaired research/validation system rather than the obsolete MVP framing.
